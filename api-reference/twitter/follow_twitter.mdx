Follow a user on behalf of the authenticated source user. Routed through the AIsa gateway at `https://api.aisa.one/apis/v1/twitter/follow_twitter`.

## Prerequisites

- An **AIsa API key**. Pass it as `aisa_api_key` in the request body.
- A one-time **OAuth authorization** for the source user account. Link your X account by calling [`POST /apis/v1/twitter/auth_twitter`](/api-reference/twitter/auth_twitter) — AIsa stores the session against your API key and uses it automatically on every write call.
- The X session must hold `follows.write`, `tweet.read`, `users.read`.

## Request body

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `aisa_api_key` | string | yes | Your AIsa API key. |
| `target_user_id` | string | yes | Numeric ID of the user to follow. |

## Response

```json
{
  "ok": true,
  "code": 200,
  "data": {
    "following": true
  }
}
```

**`data.following`** — `true` once the source user now follows the target user. `data.pending_follow` is `true` when the target is a protected account and the follow request is awaiting approval.

## Idempotency

Following a user already followed is a no-op. The endpoint still returns `200` with `data.following: true`.

## Common 4xx causes

- `400 invalid-request` — missing or non-numeric `target_user_id`.
- `403 client-forbidden` — OAuth session missing the `follows.write` scope. Re-link via [`auth_twitter`](/api-reference/twitter/auth_twitter).
- `404 resource-not-found` — target not found or not visible to the source user.

See [Error Codes](/api-reference/errors) and [Rate Limits](/api-reference/rate-limits) for more.
