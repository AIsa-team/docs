Publish a tweet on behalf of the authenticated source user. Routed through the AIsa gateway at `https://api.aisa.one/apis/v1/twitter/post_twitter`.

One endpoint covers every variant via two discriminator fields:

<CardGroup cols={2}>
  <Card title="Plain text" icon="pen">
    Send only `content`.
  </Card>
  <Card title="Reply" icon="reply">
    `type: "reply"` + `in_reply_to_tweet_id`.
  </Card>
  <Card title="Quote tweet" icon="quote-left">
    `type: "quote"` + `quote_tweet_id`.
  </Card>
  <Card title="With media" icon="image">
    Attach by pre-uploaded IDs (`media_ids`) or binary files (`multipart/form-data` with `media_files`).

## Prerequisites

- An **AIsa API key**. Pass it as `aisa_api_key` in the request body.
- A one-time **OAuth authorization** for the source user account. Link your X account by calling [`POST /apis/v1/twitter/auth_twitter`](/api-reference/twitter/auth_twitter) â€” AIsa stores the session against your API key and uses it automatically on every write call.
- The X session must hold `tweet.read`, `tweet.write`, `users.read`.

## Request fields

| Field | Type | Required | Meaning |
| --- | --- | --- | --- |
| `aisa_api_key` | string | **yes** | Your AIsa API key. |
| `content` | string | required unless posting pure media | The text of the tweet. |
| `type` | `"reply"` \| `"quote"` | no | Discriminator â€” set when replying or quoting. Omit for top-level tweets. |
| `in_reply_to_tweet_id` | string | yes when `type=reply` | Tweet being replied to. |
| `quote_tweet_id` | string | yes when `type=quote` | Tweet being quoted. |
| `media_ids` | string[] (1â€“4) | no | Pre-uploaded media IDs. |

For binary media upload, use `multipart/form-data` with a `media_files` field instead of `media_ids`. AIsa uploads the files to X for you.

## Response

On success the endpoint returns:

```json
{
  "ok": true,
  "code": 200,
  "data": {
    "tweet_id": "1346889436626259968"
  }
}
```

Use `data.tweet_id` to reference this tweet in subsequent calls (reply, quote, like).

## Common 4xx causes

- `400 invalid-request` â€” `content` missing and no media attached, or `type`/ID combination mismatch (e.g., `type: "reply"` but no `in_reply_to_tweet_id`).
- `403 client-forbidden` â€” OAuth session missing the `tweet.write` scope. Re-link via [`auth_twitter`](/api-reference/twitter/auth_twitter).
- `404 resource-not-found` â€” `in_reply_to_tweet_id` or `quote_tweet_id` doesn't exist or isn't visible to the source user.

See the full list on [Error Codes](/api-reference/errors) and [Rate Limits](/api-reference/rate-limits) for tier caps.
